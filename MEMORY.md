# 项目 Memory

## 用户长期要求

- 所有面向用户交流使用中文，不再使用旧固定前缀。
- 用户希望系统是“本地 Codex 工程 Agent”，不是只给计划，要实际推进、创建文件、跑测试、修复问题。
- 用户不懂或不想处理技术细节时，系统要提供选择题式引导。
- 用户希望项目轻量、可迁移、可审计，不要做成复杂平台。
- 用户希望结论是研究辅助，不输出确定买入/卖出，不自动交易。

## 这次用户明确指出的问题

- 当前 `book_skills/` 文档太弱，只是少量策略骨架，不满足要求。
- 需要系统性读完参考书：逐本 OCR 转 txt，通读 txt，再整理策略、观点、定量阈值。
- 强制要求：扫描版 PDF 需要全书 OCR 成 txt；未全量 OCR 时，只能标记 partial，不能宣称已完成系统性通读覆盖。
- 不能只靠文件名、关键词或少量页码生成策略。
- 需要检查每本书中的重要策略和观点是否完整覆盖。
- 每条策略、观点和阈值都要记录来源，例如书名、章节、页码或 OCR 页。
- OCR 后的文本要作为本地私有缓存，不把完整原文放进用户报告。
- 可以每本书 txt 转完后拉一个 subagent 专门整理。

## 必须避免的坑

- 不要为了让策略卡数量看起来多而编造书中没有的内容。
- 不要把低置信 OCR 内容当作高置信策略。
- 不要把低置信、否决或暂缓策略放进正式 `book_skills/strategy_cards.yaml`；这些只能进入低置信/暂缓文件。
- 不要只生成空泛原则，例如“顺势而为”“注意风险”，必须尽量提取可操作条件、反证、失效条件和定量阈值。
- 不要让技术形态单独决定最终结论。
- 不要因资金流、新闻、K线单项信号就下确定性判断。
- 不要使用付费数据源、会员网站、券商实盘接口或重型交易系统。
- 不要删除、移动、覆盖 `E:\stock\ref` 中的原始 PDF。
- 不要把 OCR 原文大段暴露在报告里。
- 不要把公开聚合公告说成官方公告原文闭环。
- 不要把 mootdx/通达信协议说成交易所直连原始逐笔或 Level-2。
- 不要把“最近 20 条日 K”直接说成已经验证“当日盘中 K 线”，除非字段和时间戳证明包含当前交易日。

## Book Skills 返工目标

每本书应生成至少以下可审计材料：

- PDF 页数和文本层/OCR 状态。
- 完整 txt 或尽可能完整 OCR txt，保存于私有缓存。
- 章节目录和页码映射。
- 逐章摘要。
- 重要策略、观点、定量阈值、反证条件、适用/不适用场景。
- A 股适配说明。
- 覆盖审计：哪些重要内容已进入 strategy cards，哪些只进低置信笔记，哪些需要人工复核。

## 用户偏好的交流方式

- 先说明能力边界，再给选择题。
- 选择题要短、明确、可执行。
- 遇到长任务时，说明当前能推进什么、需要多长、可能卡在哪里。
- 报告要有证据、最大不确定性、最强反证、下一步验证。
- 用户明确要求形成固定答复流程：先复述问题和调用能力，再给决策答案，再列输入数据流及来源，再列 book skill 来源，最后预测用户疑问并给 3-8 个选择题。
- 研究答复的标准顺序必须是：
  1. 简要复述对用户问题的理解，并说明调用了哪些能力。
  2. 给出研究判断和允许的研究分级。
  3. 说明三路输入信息流中哪些关键数据辅助决策，并标注来源。
  4. 说明引用了哪些 book skills，精确到书名、章节、策略 ID、页码线索。
  5. 说明支持证据、最大不确定性、最强反证、下一步验证。
  6. 预测用户可能继续追问的方向，给 3-8 个选择题，引导进一步探索。
